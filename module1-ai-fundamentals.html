🧠 LLMs - מה זה בכלל "מודל שפה גדול"

LLM (Large Language Model) הוא רשת נוירונים מסוג Transformer - לרוב בארכיטקטורת decoder-only - עם מיליארדי פרמטרים (משקלים נלמדים), שאומנה למשימה מתמטית פשוטה להפליא: חיזוי הטוקן הבא בהינתן כל הטוקנים שלפניו. כל היכולות ה"חכמות" שאנחנו רואים - כתיבה, קוד, נימוק - מתפתחות כתופעת-לוואי של אימון בקנה מידה עצום על המשימה הזו.

המנגנון המרכזי שמאפשר את זה הוא Self-Attention: בכל שכבה, כל טוקן "מביט" בכל הטוקנים האחרים בהקשר ומחשב כמה כל אחד מהם רלוונטי אליו כרגע (באמצעות מטריצות Query/Key/Value), ומצרף מידע בהתאם. כמה "ראשי קשב" (attention heads) פועלים במקביל ולוכדים סוגי קשרים שונים (תחביריים, סמנטיים, ארוכי-טווח), והכל נערם על פני עשרות שכבות.

האימון עצמו מתבצע במספר שלבים נפרדים: Pretraining - חיזוי הטוקן הבא על קורפוס טקסט אדיר (טריליוני טוקנים) - זה המקור העיקרי לידע וליכולת שפה. Fine-tuning/Instruction tuning - מלמד את המודל לעקוב אחר הוראות בפורמט שיחה. RLHF / Constitutional AI וטכניקות דומות - מיישרות את הפלט להעדפות אנושיות ולעקרונות בטיחות. ההבדל בין base model (שרק ממשיך טקסט בצורה גולמית) ל-instruct/chat model (כמו קלוד, שמתנהג כעוזר שיחתי) הוא בדיוק תוצאה של השלבים הנוספים האלה.

🤔 הערה אפיסטמית: השאלה האם מודל "מבין" באמת את מה שהוא כותב היא שאלה פילוסופית שנויה במחלוקת. הניסוח המדויק והזהיר: המודל לומד מבנה סטטיסטי עמוק של שפה, שמתברר כמקודד בתוכו כמות עצומה של ידע עולם ויכולות דמויות-נימוק - בלי לקבוע באופן חד-משמעי אם זו "הבנה" במובן האנושי.

מגמת Scaling Laws: במשך שנים, יכולות המודל השתפרו בצורה צפויה יחסית עם יותר פרמטרים, יותר נתונים, ויותר חישוב אימון. בשנים האחרונות נוספו מקורות שיפור נוספים שלא תלויים רק ב"גודל" - איכות הנתונים, שיטות post-training מתקדמות, וחישוב בזמן ההסקה עצמה (inference-time compute / "חשיבה" מורחבת לפני מתן תשובה).

🔬 מעבדה: הנוסחה מאחורי Self-Attention
Scaled Dot-Product Attention
Attention(Q, K, V) = softmax( (Q · K^T) / sqrt(d_k) ) · V

// Q (Query), K (Key), V (Value) - הטלות לינאריות של embedding הטוקן
// d_k - מימד וקטור ה-Key (הגורם sqrt(d_k) מייצב את הגרדיאנטים)
// softmax הופך את הציונים להתפלגות משקלים שמסכמת ל-1
// התוצאה: כל טוקן מקבל "ממוצע משוקלל" של כל הטוקנים הרלוונטיים לו
💡 בלוק טרנספורמר בודד נראה בערך כך: קלט → Self-Attention → Add&Norm → Feed-Forward → Add&Norm - וזה נערם N פעמים (עשרות שכבות במודלים גדולים) לפני הטלה סופית להתפלגות הסתברות על פני אוצר המילים כולו, בדיוק כמו שנראה בשיעור הבא על Temperature.
🔍 המחשה: לפני שהמודל "בוחר" מילה, כל טוקן קודם מביט בשאר הטוקנים (Attention). לחצו כדי להדגים סבב קשב פשוט:
לחצו כדי לראות דוגמת קשב על המשפט: "הכלב רדף אחרי החתול כי הוא רעב"

🧱 Tokenization - איך טקסט הופך למספרים

מודל שפה לא "רואה" אותיות או מילים שלמות. לפני שהטקסט מגיע למודל, רכיב בשם Tokenizer מפרק אותו ליחידות קטנות בשם tokens - לרוב באמצעות אלגוריתם כמו Byte-Pair Encoding (BPE), שבונה אוצר מילים של עשרות אלפי עד מאות אלפי "חתיכות" נפוצות מתוך קורפוס טקסט ענק.

מילים נפוצות (כמו "the" או "שלום") הופכות לרוב לטוקן בודד. מילים נדירות, שמות פרטיים, קוד תכנות או מספרים מתפצלים למספר טוקנים. שפות שאינן אנגלית - כולל עברית - נוטות "לצרוך" יותר טוקנים לאותה כמות תווים, כי אוצר המילים של רוב המודלים אומן בעיקר על טקסט אנגלי. זו לא רק סוגיה טכנית: היא משפיעה ישירות על מחיר (חיוב הוא לפי מספר טוקנים), על מהירות, ועל כמה טקסט נכנס ב-Context Window (השיעור הבא).

חוק אצבע גס: בטקסט אנגלי, טוקן אחד ≈ 4 תווים בממוצע. בעברית היחס פחות נוח ולעיתים קרוב יותר לטוקן על כל מילה או חצי מילה. חשוב גם לזכור: אין תקן אחיד - לכל משפחת מודלים יש Tokenizer משלה, כך שאותו משפט יכול להיספר אחרת ב-Claude לעומת מודלים אחרים.

🌡️ Temperature - שליטה בהתפלגות ההסתברות

בכל צעד, המודל לא "בוחר" את הטוקן הבא - הוא מחשב התפלגות הסתברויות על פני כל אוצר המילים (למשל, "80% סיכוי שהמילה הבאה היא 'שמח', 12% 'עצוב'..."). Temperature הוא פרמטר שמשנה את "חדות" ההתפלגות הזו לפני הדגימה (טכנית: הוא מחלק את ה-logits לפני הפעלת softmax).

ב-Temperature נמוך ההתפלגות נהיית חדה מאוד - כמעט כל המשקל עובר לטוקן הכי סביר, והתוצאה קרובה לדטרמיניסטית (אך לא בהכרח זהה ב-100% מהמקרים, כי יש עוד גורמי דגימה). ב-Temperature גבוה ההתפלגות משתטחת - גם טוקנים פחות סבירים מקבלים סיכוי הגון להיבחר, מה שמייצר יותר יצירתיות אך גם יותר סיכון לחוסר קוהרנטיות.

📌 דיוק חשוב: ה-API של Anthropic מקבל Temperature בטווח 0 עד 1 (לא 0 עד 2 כמו בממשקים מסוימים של ספקים אחרים). בדמו למטה השתמשנו בטווח 0-2 להמחשה חינוכית מוגזמת - כדאי לזכור את ההבדל כשעובדים עם APIs אמיתיים.

פרמטרים קרובים שכדאי להכיר: top_p (Nucleus Sampling - שוקל רק את קבוצת הטוקנים המצטברת שמגיעה לאחוז הסתברות מסוים) ו-top_k (מגביל את הדגימה ל-K הטוקנים הכי סבירים בלבד). לרוב לא משנים את כולם יחד - Anthropic ממליצה לשנות temperature או top_p, לא את שניהם בו-זמנית.

🧠 Context Window - גבול הזיכרון בפועל

Context Window הוא מספר הטוקנים המקסימלי שמודל יכול "להביט" בו בבת אחת בקריאה בודדת - system prompt + היסטוריית שיחה + ההודעה הנוכחית + מקום שמור לתשובה, הכל ביחד. זה לא "זיכרון לטווח ארוך" - ברגע שהשיחה מסתיימת, שום דבר לא נשמר אלא אם אפליקציה חיצונית שומרת ומזינה אותו מחדש בפעם הבאה.

גודל החלון משתנה בין דורות מודלים ומשפחות שונות, וגדל באופן עקבי עם הזמן - חלק מהמודלים תומכים כיום בחלונות של מאות אלפי טוקנים ואף יותר. אבל חלון גדול יותר לא פותר הכל: מחקרים מראים תופעה שנקראת "lost in the middle" - מידע שנמצא באמצע הקשר ארוך מאוד נוטה להיזכר פחות טוב מאשר מידע שבתחילת או בסוף ההקשר.

כשההקשר מתמלא, יש כמה אסטרטגיות מקובלות: חיתוך (זריקת ההודעות הישנות ביותר), סיכום (החלפת היסטוריה ארוכה בתקציר קצר), ו-RAG (השיעור הבא) - שבמקום להכניס את כל המידע לקונטקסט, שולף רק את הקטעים הרלוונטיים לשאלה הנוכחית.

🔬 מעבדה: תקציב הטוקנים של קריאה בודדת
budget (pseudo)
system_prompt_tokens
+ conversation_history_tokens
+ current_user_message_tokens
+ reserved_for_output_tokens (max_tokens)
─────────────────────────────
≤ context_window_limit   // אחרת: שגיאת "context length exceeded"
💡 שימו לב ל־max_tokens בבקשה: זה לא "כמה שהמודל צריך" אלא תקרה שאתם קובעים מראש. אם היא נמוכה מדי, התשובה תיחתך גם אם עדיין נשאר מקום בחלון ההקשר.

ניהול הקשר ברמת האפליקציה: כשהשיחה מתארכת, יש כמה תבניות נפוצות מעבר לחיתוך גס - Sliding Window (שמירת ה-N ההודעות האחרונות בלבד), Summarization Chain (סיכום אוטומטי של הודעות ישנות לפסקה קצרה במקום למחוק אותן לגמרי), ו-Hierarchical Memory (שילוב של זיכרון קצר-טווח בהקשר עצמו + זיכרון ארוך-טווח חיצוני שנשלף לפי הצורך דרך RAG).

Prompt Caching - טכניקה אמיתית וזמינה בפועל ב-API: כאשר חלק גדול מהפרומפט חוזר על עצמו בין קריאות (system prompt ארוך, מסמך רקע קבוע), אפשר לסמן אותו כ"נקודת cache" - השרת שומר את מצב החישוב הפנימי (Key/Value) של אותו קידומת, וקריאות עוקבות שמשתמשות באותה קידומת בדיוק נהנות מזמן עיבוד וממחיר נמוכים משמעותית, במקום לעבד את אותו טקסט מחדש בכל פעם.

🍽️ API - החוזה הפורמלי מול המודל

API (Application Programming Interface) הוא חוזה מוגדר: כתובת (endpoint), פורמט בקשה, פורמט תשובה, ואימות הרשאות - שמאפשר לתוכנה שלכם "לדבר" עם שרתי הספק בלי שתריצו את המודל בעצמכם. אנלוגיית המלצר נכונה ברמת העיקרון, אבל ה-API האמיתי מוסיף שכבות: מפתח API (אישור זהות וחיוב), גרסת API (headers כמו anthropic-version), הגבלות קצב (rate limits), קודי שגיאה סטנדרטיים (400, 401, 429, 500...), ו-ניסיונות חוזרים (retries) בכשלים זמניים.

בקשה טיפוסית כוללת: שם מודל, מערך messages עם תפקידים (user/assistant), ופרמטרים כמו temperature ו-max_tokens. התשובה כוללת בלוקי תוכן, stop_reason (למה המודל הפסיק - סיים תשובה? הגיע למגבלת טוקנים? ביקש להשתמש בכלי?), ואת אובייקט ה-usage שראינו בשיעור על Tokens.

💡 שימו לב שהמפתח (x-api-key) נשלח כ-header, לא בגוף הבקשה - כלל אצבע אבטחתי: סודות לא נכנסים ל-body או לכתובת ה-URL, כי אלו נוטים להישמר בלוגים.

Streaming: בממשקי צ'אט אמיתיים המודל לא "נעלם" לכמה שניות ואז מחזיר הכל בבת אחת - הבקשה נשלחת עם "stream": true, והשרת מחזיר את התשובה כזרם אירועים (Server-Sent Events) שמגיעים טוקן-טוקן, כדי שהמשתמש יראה טקסט מופיע בזמן אמת במקום לחכות לכל התשובה.

התמודדות עם כשלים: קריאות API אמיתיות נכשלות מדי פעם - עומס זמני (429), תקלת שרת (5xx), או timeout ברשת. התבנית הסטנדרטית היא exponential backoff with jitter: בכל ניסיון חוזר מחכים זמן שגדל אקספוננציאלית (1s, 2s, 4s...) בתוספת רעש אקראי קטן.

🔍 RAG - שליפה מבוססת-עדות לפני יצירה

Retrieval-Augmented Generation הוא Pipeline בן כמה שלבים: (1) חיתוך מסמכים לקטעים (chunks), (2) הטמעה (embedding) של כל קטע לוקטור מספרי, (3) שמירה במסד וקטורים (Vector DB / אינדקס ANN כמו HNSW), (4) בזמן שאלה - הטמעת השאלה עצמה, (5) שליפה של ה-K הקטעים הכי דומים (לרוב לפי Cosine Similarity), (6) הזרקת הקטעים שנשלפו כהקשר לתוך הפרומפט, (7) המודל מייצר תשובה המבוססת (מקווים) על הטקסט שנשלף.

למה זה חשוב: זו דרך לתת למודל גישה למידע עדכני או פרטי בלי לאמן אותו מחדש, ולהפחית הזיות (Hallucinations) על ידי "עיגון" התשובה בעדות ממשית. אבל יש כאן נקודת תורפה קריטית: איכות התשובה חסומה על ידי איכות השליפה - "garbage in, garbage out". אם החיתוך גרוע או ההטמעה לא מדויקת, המודל יקבל הקשר לא רלוונטי ועדיין "יאמין" שהוא נכון.

🔬 מעבדה: הנוסחה שמאחורי "החיפוש החכם"
Cosine Similarity
similarity(A, B) = (A · B) / (‖A‖ × ‖B‖)

// דוגמה מספרית עם שני וקטורים דו-ממדיים:
A = [0.9, 0.1]   // "כלב רץ בפארק"
B = [0.8, 0.2]   // "כלבלב משחק בחוץ"
A·B = 0.9*0.8 + 0.1*0.2 = 0.74
‖A‖ = 0.906, ‖B‖ = 0.825
similarity ≈ 0.74 / (0.906*0.825) ≈ 0.99  →  קרובים מאוד במשמעות
💡 ערך קרוב ל-1 = כמעט זהים במשמעות. ערך קרוב ל-0 = לא קשורים. ערך שלילי (בטווח -1 עד 1) = מנוגדים סמנטית. זה בדיוק המנוע מאחורי "החיפוש" בדמו למטה.

איפה שומרים את הוקטורים? במסד וקטורים (Vector Database) - כלים ייעודיים כמו Pinecone, Weaviate, Qdrant או Chroma, לעומת הרחבות וקטוריות למסדי נתונים קיימים כמו pgvector עבור Postgres. הבחירה תלויה בקנה מידה, בדרישות latency, ובכמה חשוב לכם לא להוסיף עוד מערכת נפרדת לתחזק.

אסטרטגיית חיתוך (Chunking) משפיעה ישירות על איכות השליפה: chunks קטנים מדי מאבדים הקשר; chunks גדולים מדי "מדללים" את האות הרלוונטי בתוך רעש. פרקטיקה נפוצה: chunks בגודל 200-500 טוקנים עם overlap (חפיפה) של 10-20% ביניהם.

📤 שלב 1: "העלו" קובץ (מדמה חיתוך + הטמעה לוקטורים)
❓ שלב 2: שאלו שאלה (מדמה שליפה לפי דמיון וקטורי)

🗺️ Embeddings - משמעות כגיאומטריה

Embedding הוא וקטור מספרי צפוף (בפועל - לרוב מאות עד אלפי ממדים, לא 3 כמו בדמו למטה) שמיוצר על ידי רשת נוירונים מאומנת, כך שקלטים בעלי משמעות דומה ממופים לנקודות קרובות במרחב הווקטורי. מרחק בין נקודות נמדד לרוב ב-Cosine Similarity או במרחק אוקלידי.

איך זה מתאמן, בקצרה: Contrastive Learning - במהלך האימון, המודל נענש כשהוא ממקם זוגות "דומים" רחוק, ומתוגמל כשהוא ממקם אותם קרוב, ולהיפך עבור זוגות "שונים". מודלי Embedding הם בדרך כלל מודלים נפרדים ומתמחים - לא אותו מודל גנרטיבי שעונה לכם על שאלות.

🔬 מעבדה: חישוב דמיון אמיתי בין שלוש מילים
toy vectors (2D, להמחשה)
חתול  = [0.90, 0.85]
כלב   = [0.88, 0.80]
אוטו  = [0.10, 0.95]

similarity(חתול, כלב) ≈ 0.999   →  קרובים מאוד (שתיהן חיות מחמד)
similarity(חתול, אוטו) ≈ 0.79   →  רחוקים משמעותית פחות
💡 בעולם האמיתי הווקטורים ארוכים בהרבה, אבל העיקרון זהה לחלוטין.

טרייד-אוף מימדיות: מודל embedding עם יותר מימדים (למשל 3072 לעומת 384) לרוב לוכד יותר ניואנס סמנטי, אבל עולה יותר באחסון ובזמן חיפוש. השוואת איכות בין מודלי embedding נעשית בדרך כלל מול benchmarks סטנדרטיים (כמו MTEB).

Matryoshka Embeddings - טכניקה חדשה יחסית: מודלים מסוימים מאומנים כך שאפשר "לקצץ" (truncate) את הוקטור למימדים פחותים ועדיין לקבל דמיון סמנטי שמיש, עם ירידה הדרגתית באיכות ולא קריסה מוחלטת.

🔬 מעבדה: קיצוץ מימדים (Matryoshka) להמחשה
truncation example
full_vector (1536 dims)  → similarity(A,B) = 0.912
truncated to 512 dims    → similarity(A,B) = 0.905   // כמעט זהה
truncated to 128 dims    → similarity(A,B) = 0.861   // ירידה מתונה
truncated to 32 dims     → similarity(A,B) = 0.640   // ירידה משמעותית
💡 שימו לב לירידה ה"חלקה" - זה ההבדל בין matryoshka embeddings לחיתוך מקרי של מודל רגיל.
🔄 שימו לב: חתול וכלב קרובים (שניהם חיות), אבל אוטו רחוק יותר במרחב:
🐱 חתול
🐶 כלב
🚗 אוטו
המרחב מסתובב לבד - שימו לב שהמרחק בין המילים נשמר מכל זווית 👀

🔁 Agentic Loop - מלולאת חשיבה ליישות פועלת

לולאה אג'נטית (מוכרת גם בשם דפוס ReAct - Reasoning + Acting) עובדת כך: Observe (המודל רואה את מצב השיחה הנוכחי) → Think (מנמק מה השלב הבא) → Act (בוחר ומפעיל כלי - tool call) → Observe את תוצאת הכלי → וחוזר חלילה, עד לתנאי עצירה.

זו בדיוק הארכיטקטורה שמפעילה בפועל את Claude Code ואת Claude Agent SDK: כל איטרציה היא קריאת API בודדת שעשויה להחזיר בלוק מסוג tool_use; ה-harness מבצע את הפעולה בפועל, ומזין את תוצאת הכלי (tool_result) בחזרה כהודעה בהיסטוריה עבור האיטרציה הבאה.

מוקשים אמיתיים: לולאות אינסופיות, שימוש שגוי בכלים, וההיסטוריה שגדלה בכל איטרציה. זו בדיוק הסיבה ש-Subagents (מודול 2) קיימים: הם "מכילים" גידול הקשר בלתי נשלט עבור תת-משימות מסוימות.

🔬 מעבדה: פסאודוקוד של הלולאה בפועל
pseudo-python
messages = [system_prompt, user_message]
done = False
while not done:
    response = model.call(messages)
    if response.stop_reason == "tool_use":
        result = run_tool(response.tool_call)
        messages.append(tool_result(result))
    else:
        done = True
        final_answer = response.text
💡 העצירה קורית כי stop_reason כבר לא tool_use. אם המודל תמיד מחזיר tool_use, הלולאה לא תיעצר לבד - מכאן חשיבות ה-max_turns וה-timeout.
