🧩 Skills - יחידות יכולת הניתנות לגילוי אוטומטי

Skill הוא חבילת הוראות עצמאית - תיקייה שמכילה קובץ SKILL.md עם YAML frontmatter ותוכן Markdown, השוכנת ב-.claude/skills/<name>/ (ברמת פרויקט) או ~/.claude/skills/<name>/ (ברמה אישית). השדה הקריטי הוא description: זה בדיוק מה שקלוד קורא כדי להחליט אם הבקשה שלכם "מתאימה" לסקיל הזה - בלי שתצטרכו לקרוא לו בשם במפורש.

בניגוד לפרומפט חד-פעמי, Skill הוא קבוע וניתן לשימוש חוזר: הוא נטען לתוך ההקשר רק כאשר הבקשה תואמת לתיאור שלו, מה שהופך אותו לפתרון סקיילבילי - אפשר להוסיף עשרות סקילים בלי "להציף" את קלוד בכל ההוראות בכל שיחה.

שדות מתקדמים: allowed-tools (מגביל אילו כלים הסקיל מורשה להשתמש בהם), disable-model-invocation (אם true - רק המשתמש יכול להפעיל את הסקיל במפורש), ו-context: fork (מריץ את הסקיל כתת-סוכן מבודד).

🔬 מעבדה: קובץ SKILL.md אמיתי
.claude/skills/pdf-report/SKILL.md
---
name: pdf-report
description: Generate formatted PDF reports from markdown
  templates. Use when the user asks to create, compile,
  or export a report to PDF.
allowed-tools:
  - Read
  - Write
  - Glob
disable-model-invocation: false
---
# PDF Report Generator
You are a report generation specialist. When invoked:
1. Read the user's markdown content or template
2. Enhance formatting, structure, and visual hierarchy
3. Generate a polished PDF with a cover page, TOC, and
   consistent styling
💡 השם בשדה name חייב להתאים לשם התיקייה עצמה. ה-description ארוך ומפורט בכוונה - description עמום יגרום לסקיל להיות מופעל באופן לא עקבי.
✍️ כתבו בקשה ותראו איזה סקיל "נדלק" לפי ה-description שלו:

🖐️ Tools - הממשק הפורמלי בין המודל לעולם

מבחינה טכנית, "כלי" הוא פונקציה בעלת שם, תיאור, וסכימת קלט (JSON Schema) שמוגדרת לפני הקריאה למודל. כשהמודל "מחליט" להשתמש בכלי, הוא לא מריץ שום דבר בעצמו - הוא פשוט מחזיר בלוק תוכן מסוג tool_use עם שם הכלי וארגומנטים. ה-harness הוא זה שבפועל מריץ את הפעולה, ומחזיר את התוצאה כבלוק tool_result בהודעה הבאה.

קלוד קוד מגיע עם כלים מובנים כמו Read, Write/Edit, Bash, ו-WebSearch/WebFetch. כלים נוספים יכולים להגיע דרך שרתי MCP (השיעור הבא) - כלים מובנים תמיד זמינים; כלי MCP הם תוספת אופציונלית שדורשת חיבור מפורש.

🔬 מעבדה: איך קריאת כלי נראית בפרוטוקול עצמו
tool_use content block
{
  "type": "tool_use",
  "id": "toolu_01A...",
  "name": "Read",
  "input": { "file_path": "/project/README.md" }
}

// ואז ה-harness מחזיר בהודעה הבאה:
{
  "type": "tool_result",
  "tool_use_id": "toolu_01A...",
  "content": "# Welcome to the project\n..."
}
💡 שרשרת tool_use → tool_result היא בדיוק מה שראינו כפסאודוקוד בשיעור Agentic Loop (מודול 1) - זה לא מטאפורה, זה הפורמט המילולי שמופיע ב-messages array.
🖱️ לחצו על כלי ותראו מה קלוד "עושה" איתו בפועל:
בחרו כלי כדי לראות אותו בפעולה 👆

🌉 MCP - פרוטוקול תקני לחיבור כלים חיצוניים

Model Context Protocol הוא תקן פתוח בארכיטקטורת client-server: קלוד הוא ה-client, ו-שרת MCP הוא תהליך נפרד שיודע "לתרגם" בין הפרוטוקול הגנרי לבין ה-API הספציפי של שירות חיצוני (Slack, GitHub, מסד נתונים פנימי).

שרתי MCP מוגדרים בקובץ .mcp.json. שרת מקומי (stdio) מוגדר עם command ו-args. שרת מרוחק (HTTP) מוגדר עם url. שני הסוגים יכולים לקבל משתני סביבה (env) עם תמיכה בהרחבה מהצורה ${VAR}, כדי שסודות לא יישמרו ישירות בקובץ ה-JSON.

🔬 מעבדה: קובץ .mcp.json אמיתי
.mcp.json
{
  "mcpServers": {
    "github": {
      "command": "npx",
      "args": ["@modelcontextprotocol/server-github"],
      "env": { "GITHUB_TOKEN": "${GITHUB_TOKEN}" },
      "type": "stdio"
    },
    "remote-analytics": {
      "url": "https://api.example.com/mcp",
      "headers": { "Authorization": "Bearer ${AUTH_TOKEN}" },
      "type": "http"
    }
  }
}
💡 שימו לב לשני הדפוסים באותו קובץ: command/args לשרת מקומי מול url/headers לשרת מרוחק.

🕵️ Subagents - בידוד הקשר לצורך האצלה

Subagent הוא מופע קלוד נפרד ומבודד - עם system prompt משלו, רשימת כלים מוגבלת (tools / disallowedTools), ולעיתים אפילו מודל אחר (model). כשהסוכן הראשי מאציל משימה עצמאית, ה-subagent מבצע את כל העבודה בהקשר משלו, ומחזיר לסוכן הראשי רק את התוצאה הסופית.

זה פותר ישירות את בעיית גדילת ההקשר שראינו ב-Agentic Loop (מודול 1): אם משימת מחקר דורשת 20 חיפושי אינטרנט, כל 20 התוצאות "נחיות" בתוך ה-subagent, ולא מציפות את חלון ההקשר הראשי. זה גם מאפשר התמחות.

🔬 מעבדה: הגדרת subagent אמיתית
.claude/agents/code-reviewer.md
---
name: code-reviewer
description: Expert code review specialist. Use when
  reviewing pull requests or architectural changes.
tools:
  - Read
  - Grep
  - Glob
model: sonnet
disallowedTools:
  - Write
  - Edit
  - Bash
permissionMode: read-only
---
# Code Review Specialist
You are a senior engineer performing code reviews.
Check for bugs, performance issues, security gaps,
and style violations. Provide actionable feedback.
💡 שימו לב ל-disallowedTools: זו הגנה מפורשת - גם אם מישהו ינסה לשכנע את ה-subagent הזה לכתוב קובץ, הוא פיזית לא יכול.

⚡ Slash Commands - הפעלה מפורשת מול הפעלה אוטומטית

פקודת סלאש היא הפעלה יזומה של המשתמש (מתחילה ב-/). ההבדל המהותי מ-Skill: פקודה מוגדרת בקובץ שטוח ב-.claude/commands/<name>.md ומופעלת רק על ידי הקלדה מפורשת. Skill, לעומת זאת, נמצא בתיקייה עשירה יותר ויכול להיות מופעל גם על ידי /שם וגם באופן אוטומטי כשהתיאור שלו תואם לבקשה.

שתי תוספות מתקדמות: /goal - מגדיר מצב-סיום ניתן לאימות שקלוד ממשיך לעבוד לקראתו לבד בין תורות; ו-/loop - מריץ קלוד במחזורים חוזרים, אופציונלית עד תנאי עצירה (until). שני אלה הופכים אינטראקציה בודדת לתהליך עבודה מתמשך - הבסיס לשיעור הבא.

🔬 מעבדה: פקודה מותאמת אישית + /goal ו-/loop
.claude/commands/deploy.md
---
name: deploy
description: Deploy the current branch to production
---
# Deploy
Before deploying: run tests, check for uncommitted
changes, confirm target environment, then execute
the deployment script.
שילוב /goal + /loop
/goal פרוסים ל-staging וכל הבדיקות עוברות בהצלחה
/loop until כל הבדיקות עוברות
💡 שימו לב לניסוח ה-goal: "כל הבדיקות עוברות בהצלחה" הוא מצב הניתן לאימות.

🛡️ Hooks - שכבת אכיפה דטרמיניסטית

Hook הוא סקריפט שרץ אוטומטית בנקודות חיים מוגדרות - PreToolUse (לפני הרצת כלי), PostToolUse (אחרי סיום הכלי), UserPromptSubmit, Stop ועוד. ה-hook מקבל JSON דרך stdin, ושולט בהתנהגות דרך קוד היציאה: 0 מאשר, 2 חוסם לחלוטין.

ההבדל המהותי מ-Skill או מ-System Prompt: hook הוא דטרמיניסטי - הוא אוכף בקוד ברמת התשתית, ולכן לא ניתן לעקוף אותו באמצעות שכנוע (Prompt Injection).

🔬 מעבדה: hook אמיתי שחוסם rm -rf
.claude/settings.json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [{
          "type": "command",
          "command": "bash -c 'echo \"$CLAUDE_TOOL_INPUT\" | grep -qE \"rm -rf\" && exit 2 || exit 0'"
        }]
      }
    ]
  }
}
💡 ה-matcher קובע על אילו כלים ה-hook חל. כאן: כל פקודת Bash נבדקת מול תבנית מסוכנת לפני שהיא מגיעה למעטפת.
🚦 קלוד רוצה לבצע פעולה - מה יגיד ה-hook?
🛡️
קלוד רוצה: "להריץ Bash: rm -rf ./old_files" 🗑️ - לאשר (exit 0) או לחסום (exit 2)?

📦 Plugins - אריזה, גרסאות והפצה

Plugin לא מוסיף יכולת חדשה מבחינה טכנית - כל מה שהוא מכיל (skills, hooks, MCP servers, subagents) ניתן להגדרה בנפרד תחת .claude/. מה ש-plugin כן מוסיף הוא גבול פורמלי: שם, מספר גרסה, ומניפסט (.claude-plugin/plugin.json), שהופכים אוסף הגדרות אישי ליחידה ניתנת להפצה, גרסאות, והתקנה אחידה.

מבנה תיקיות סטנדרטי: skills/, agents/, commands/, .mcp.json, ו-hooks/ - כל אלה מתגלים אוטומטית. השדה החובה היחיד למניפסט מינימלי הוא name.

🔬 מעבדה: plugin.json ומבנה תיקיות אמיתי
.claude-plugin/plugin.json
{
  "name": "document-tools",
  "version": "2.1.0",
  "description": "PDF, DOCX, and PPTX generation tools",
  "author": { "name": "Sarah Chen", "email": "sarah@co.com" },
  "license": "MIT"
}
מבנה תיקיות

⏰ סוכנים אוטונומיים - עבודה ללא לולאת שאלה-תשובה

עד כה כל השיעורים הניחו מישהו שמקליד ומחכה לתשובה. סוכנים אוטונומיים הופכים את זה: אתם מגדירים תזמון (cron) או יעד (/goal), וקלוד מתעורר, מבצע את המשימה בעצמו, ומדווח. זו בדיוק אותה לולאה אג'נטית שלמדנו במודול 1, רק שהטריגר לאיטרציה הראשונה הוא שעון ולא הודעת משתמש.

מנגנונים מעשיים: משימות מתוזמנות (cron expression), /loop שרץ במחזורים חוזרים עד תנאי עצירה, ו-/goal שממשיך אוטומטית בין תורות עד שהמצב הרצוי מתקיים.

🔬 מעבדה: תזמון בפועל
cron expression לדוגמה
0 8 * * *   → בכל יום בשעה 08:00
0 9 * * 1-5 → בימי חול בלבד, בשעה 09:00
💡 בדמו למטה כל "מחזור" מייצג ריצה מלאה של הלולאה האג'נטית.
