🛠️ Tool Use / Function Calling

Function Calling הופך את המודל ממחולל טקסט בלבד למערכת שיכולה לפעול בעולם האמיתי. מגדירים למודל רשימת "כלים" זמינים (שם, תיאור, סכימת קלט), והמודל מחליט בעצמו מתי ואיזה כלי לקרוא כדי לענות על הבקשה - במקום שאתם תכתבו את כל הלוגיקה מראש.

הזרימה בפועל: (1) שולחים בקשה + הגדרת tools[], (2) המודל מחזיר לא טקסט אלא בלוק tool_use עם שם הכלי והפרמטרים, (3) הקוד שלכם מריץ את הפונקציה המתאימה בפועל (דיספצ'ר), (4) שולחים את תוצאת הביצוע (tool_result) בחזרה למודל כדי שהוא ינסח תשובה סופית למשתמש.

נקודה קריטית: המודל לא מבצע את הפעולה בעצמו - הוא רק "מבקש" לבצע אותה. הקוד שלכם הוא זה שבפועל קורא למסד הנתונים, שולח מייל, או מבטל הזמנה. זה גם נקודת האבטחה הקריטית - תמיד לוודא הרשאות לפני ביצוע פעולה שהמודל ביקש.

📸 צילום מסך אמיתי - קוד שרץ בפועל
דיספצ'ר אמיתי שמקבל tool_use block מדומה ומריץ בפועל את הפונקציה המתאימה (cancel_order) מול "מסד נתונים" בזיכרון.

🌊 Streaming, Latency ו-Fallbacks

בstreaming, במקום לחכות לתשובה מלאה לפני שמציגים כלום, המודל שולח את התשובה chunk-אחר-chunk (בדיוק כמו שנראה בצ'אט של Claude עצמו) - וה-UI מציג אותם מיד עם הגעתם. זה לא משנה את התוכן הסופי, אבל משפר דרמטית את חוויית המשתמש הנתפסת, במיוחד בתשובות ארוכות.

Latency - זמן התגובה - משתנה בין קריאות, ולעיתים שרתי המודל עמוסים או נופלים זמנית. תכנון UX טוב כולל: אינדיקציית טעינה ברורה, timeout סביר עם retry אוטומטי (עם exponential backoff - המתנה גדלה בין ניסיונות), וfallback - תגובת ברירת מחדל הגיונית אם כל הניסיונות נכשלו, במקום מסך שגיאה גולמי.

בפרודקשן אמיתי: תמיד עוטפים קריאות API ב-try/except עם retry logic, מגדירים timeout מפורש (לא סומכים על ברירת המחדל), ומתעדים (logging) כשלים כדי לזהות דפוסי בעיה - בדיוק כמו כל קריאת רשת חיצונית אחרת במערכת פרודקשן.

📸 צילום מסך אמיתי - קוד שרץ בפועל
קוד שצורך אירועי stream בזמן-אמת ומרכיב את הטקסט הסופי chunk-אחר-chunk, בדיוק כפי שקורה בפועל ב-server-sent events.

🌐 ריבוי ספקי מודלים

לכל ספק API (Anthropic, OpenAI, Google Gemini, OpenRouter, Together AI) יש פורמט payload קצת שונה - איך system prompt מוגדר, איך מבנה ה-content נראה, איך קוראים לפרמטרים. OpenRouter שווה הכרות מיוחדת: הוא נותן גישה למגוון עצום של מודלים (כולל מודלים פתוחים) דרך API אחיד יחיד - שימושי לפרוטוטייפינג מהיר בלי לנהל כמה מפתחות API.

כשבונים מוצר אמיתי, כדאי לבנות שכבת הפשטה (abstraction layer) - קוד שממיר בקשה "אחידה" משלכם לפורמט הספציפי של כל ספק. זה מאפשר להחליף ספק (או לבדוק כמה ספקים במקביל) בלי לשכתב את כל הלוגיקה העסקית שסביב הקריאה.

למה בכלל להחליף ספק: עלות (מחירים שונים למודל דומה), ביצועים (latency שונה), יכולות ייחודיות (context window ארוך יותר, תמיכה במולטימודל), או פשוט זמינות (redundancy - אם ספק אחד נופל, לעבור לאחר אוטומטית).

📸 צילום מסך אמיתי - קוד שרץ בפועל
שכבת הפשטה אמיתית שממירה בקשה אחידה (UnifiedRequest) לפורמט payload ספציפי של Anthropic, OpenAI, ו-Gemini - שלושה מבנים שונים לגמרי.

✂️ Chunking - חיתוך מסמכים ל-RAG

לפני שאפשר לחפש במסמכים ארוכים (RAG - מודול 1), צריך לחתוך אותם לchunks - קטעים קטנים שכל אחד הופך לוקטור embedding נפרד. הגודל האידיאלי הוא טרייד-אוף: chunks קטנים מדי מאבדים הקשר, chunks גדולים מדי "מדללים" את האיתות הסמנטי ומקשים על התאמה מדויקת לשאילתה.

אסטרטגיות נפוצות: חיתוך לפי מספר תווים/טוקנים קבוע, חיתוך לפי גבולות משפט/פסקה (שומר על שלמות רעיונית), וחיתוך לפי מבנה המסמך (כותרות, סעיפים - שימושי למסמכים משפטיים/טכניים מובנים).

Overlap - חפיפה בין chunks סמוכים (למשל המשפט האחרון של chunk אחד חוזר בתחילת הבא) - מונעת מצב שבו מידע קריטי "נקטע" בדיוק בגבול בין שני chunks ואף אחד מהם לבדו לא נותן הקשר מלא.

📸 צילום מסך אמיתי - קוד שרץ בפועל
אלגוריתם chunking אמיתי לפי משפטים עם overlap, שפועל על מסמך מדיניות אמיתי ומפיק chunks עם חפיפה ביניהם.

🔍 Vector Search, Hybrid Search ו-Reranking

מסדי נתונים וקטוריים (Pinecone, Weaviate, Chroma, Qdrant) מאחסנים embeddings ומאפשרים חיפוש מהיר של "השכנים הקרובים ביותר" (Nearest Neighbor) לפי מדד דמיון כמו Cosine Similarity. חיפוש וקטורי לבדו תופס דמיון סמנטי (משמעות), אבל לפעמים "מפספס" התאמות מילוליות מדויקות.

Hybrid Search משלב חיפוש וקטורי (סמנטי) עם חיפוש מילות-מפתח קלאסי (BM25/keyword) - נותן ציון משוקלל שמשלב את שני העולמות, ולעיתים תופס תוצאות שכל שיטה לבד הייתה מפספסת.

Reranking - שלב נוסף אחרי השליפה הראשונית: מודל ייעודי (או היוריסטיקה) מדרג מחדש את התוצאות המובילות לפי רלוונטיות עדינה יותר, לפני שמזינים אותן בפועל למודל השפה. זה משפר משמעותית את איכות ה-context שמוזן ל-RAG, במיוחד כשהשליפה הראשונית מחזירה הרבה מועמדים "כמעט נכונים".

📸 צילום מסך אמיתי - קוד שרץ בפועל
חיפוש וקטורי אמיתי עם numpy (cosine similarity), ולאחריו שלב reranking שמשלב חפיפת מילות מפתח - שים לב איך סדר התוצאות משתנה בפועל.

🏛️ ארכיטקטורת RAG מקצה לקצה

עכשיו נחבר את כל השלבים (מודול 1 + השיעורים הקודמים כאן) לפייפליין RAG שלם. שלב הכנה (offline): מסמכים → chunking → embedding לכל chunk → אחסון במסד נתונים וקטורי. זה קורה פעם אחת (או מתעדכן תקופתית) לפני שמשתמש אמיתי שואל שאלה.

שלב שאילתה (online, בזמן-אמת): שאלת המשתמש → embedding לשאלה → vector search (+hybrid אם רלוונטי) → reranking → הרכבת prompt שמזריק את ה-chunks הרלוונטיים כהקשר → קריאת LLM עם ה-context המועשר → תשובה סופית למשתמש.

💡 טעות נפוצה: להזריק יותר מדי chunks "ליתר ביטחון". יותר context לא תמיד טוב יותר - chunks לא רלוונטיים "מדללים" את תשומת הלב של המודל ועלולים להוריד דיוק. עדיף 3-5 chunks באמת רלוונטיים (אחרי reranking) מאשר 20 chunks גולמיים.
