📄 HTML סמנטי ונגישות

HTML סמנטי משתמש בתגיות שמתארות משמעות, לא רק מראה: <nav>, <article>, <button> במקום <div> גנרי לכל דבר. זה משפר גם SEO (מנועי חיפוש מבינים את מבנה הדף) וגם נגישות (קוראי מסך יודעים "זה ניווט", "זה כפתור").

נגישות (Accessibility, a11y) אינה "תוספת נחמדה" - היא דרישה בסיסית. שני כללים קריטיים: כל <input> צריך <label for="..."> מקושר (כדי שקורא מסך יידע מה השדה), וכל <img> צריך alt שמתאר את התוכן (או alt="" ריק במפורש אם התמונה דקורטיבית בלבד).

טפסים (<form>) הם המקום שבו נגישות משפיעה הכי הרבה על שימושיות בפועל - טופס לא נגיש פשוט לא ניתן למילוי עבור חלק מהמשתמשים. הרגל טוב: לבדוק כל טופס עם מקלדת בלבד (בלי עכבר) - אם אי אפשר לנווט ולמלא אותו רק עם Tab ו-Enter, יש בעיית נגישות.

📸 צילום מסך אמיתי - קוד שרץ בפועל
בודק נגישות אמיתי (עם jsdom) שרץ על טופס HTML אמיתי ומזהה תמונה בלי alt.
🎯 אתגר עצמאי - נסו לפני שתציצו בפתרון
בנו טופס הרשמה עם 3 שדות (שם, אימייל, סיסמה) שעובר את כל בדיקות הנגישות:
  • כל שדה עם <label for="..."> מקושר נכון
  • שדה הסיסמה עם type="password" ו-autocomplete="new-password"
  • כפתור שליחה עם טקסט ברור (לא ריק, לא רק אייקון)
כתבו את הקוד בעורך משלכם, ובדקו ניווט מקלדת בלבד לפני שתמשיכו.
רמז: ל-label יש תכונת for שחייבת להיות זהה בדיוק ל-id של ה-input המתאים.
<form>
  <label for="fname">שם מלא</label>
  <input id="fname" name="fname" type="text" required>

  <label for="femail">אימייל</label>
  <input id="femail" name="femail" type="email" required>

  <label for="fpass">סיסמה</label>
  <input id="fpass" name="fpass" type="password"
         autocomplete="new-password" required>

  <button type="submit">הרשמה</button>
</form>

📐 CSS מודרני - Flexbox ו-Grid

Flexbox מסדר items בציר יחיד (שורה או עמודה) ומתאים במיוחד לניווטים, כרטיסים בשורה, וריווח אוטומטי בין רכיבים. flex-grow קובע כמה מהשטח הפנוי כל item "תופס" יחסית לאחרים - זה בדיוק מה שקורה מאחורי הקלעים כשהדפדפן "מחשב" רוחב סופי לכל אלמנט.

Grid מסדר items בשני צירים בו-זמנית (שורות ועמודות) - מתאים לפריסות עמוד שלמות (header/sidebar/main/footer). ב-Grid מגדירים את המבנה מראש (grid-template-columns) והפריטים "נשתלים" לתוכו, לעומת Flexbox שבו הפריסה נגזרת מתוכן הפריטים עצמם.

כלל אצבע מעשי: Flexbox לרכיב בודד (תפריט, שורת כפתורים, כרטיס), Grid לפריסת עמוד כוללת. Responsive - שימוש ב-media queries או ב-grid-template-columns: repeat(auto-fit, minmax(200px, 1fr)) מאפשר לפריסה להסתגל אוטומטית לרוחב מסך בלי לכתוב breakpoints ידניים לכל מקרה.

📸 צילום מסך אמיתי - קוד שרץ בפועל
חישוב אמיתי של אלגוריתם חלוקת השטח ב-Flexbox - בדיוק הלוגיקה שהדפדפן מריץ כשהוא מציג flex-grow בפועל.
🎯 אתגר עצמאי - נסו לפני שתציצו בפתרון
בנו פריסת עמוד עם CSS Grid: header ברוחב מלא למעלה, sidebar קבוע (200px) בצד, תוכן ראשי גמיש שממלא את השאר, ו-footer ברוחב מלא למטה. השתמשו ב-grid-template-areas לקריאות מקסימלית.
רמז: grid-template-areas מאפשר "לצייר" את הפריסה כמחרוזת טקסט - כל תא מייצג אזור.
.layout {
  display: grid;
  grid-template-columns: 200px 1fr;
  grid-template-rows: auto 1fr auto;
  grid-template-areas:
    "header header"
    "sidebar main"
    "footer footer";
  min-height: 100vh;
}
.header { grid-area: header; }
.sidebar { grid-area: sidebar; }
.main { grid-area: main; }
.footer { grid-area: footer; }

⚡ JavaScript מודרני - Async/Await

JavaScript מריץ קוד ב-thread יחיד, אבל מטפל בפעולות "איטיות" (רשת, טיימרים) בלי לחסום את שאר התוכנית, באמצעות Event Loop. Promise מייצג ערך שיהיה זמין בעתיד; async/await הוא תחביר נוח יותר לעבוד עם Promises בלי "פירמידת callbacks".

הבדל קריטי בביצועים: await בתוך לולאה רגילה מריץ בקשות ברצף (כל אחת מחכה לקודמת) - איטי. Promise.all() מריץ אותן במקביל - מהיר משמעותית כשהבקשות לא תלויות זו בזו. זו טעות ביצועים נפוצה מאוד אצל מתחילים.

Destructuring (const {'{'} name, active {'}'} = user) ו-array methods מודרניים (filter, map, reduce) הופכים קוד לקצר וקריא יותר מלולאות for מסורתיות. fetch() הוא ה-API הסטנדרטי המובנה לקריאות HTTP בדפדפן - מחזיר Promise שצריך await.

📸 צילום מסך אמיתי - קוד שרץ בפועל
מדידת זמן אמיתית: טעינה סדרתית (256ms) מול Promise.all מקבילי (51ms) - אותה תוצאה, פי 5 יותר מהיר.
🎯 אתגר עצמאי - נסו לפני שתציצו בפתרון
כתבו פונקציה אסינכרונית fetchAllOrRetry(urls) שמנסה להביא כמה URLs במקביל, ואם אחד נכשל - מנסה אותו שוב פעם אחת נוספת לפני שהיא מוותרת (לא מפילה את כל הפעולה בגלל כישלון בודד).
רמז: Promise.allSettled (לא all) לא נכשל כשפרומיס אחד נדחה - זה מה שמאפשר retry סלקטיבי.
async function fetchWithRetry(url) {
  try { return await fetch(url); }
  catch (e) { return await fetch(url); } // ניסיון שני
}
async function fetchAllOrRetry(urls) {
  const results = await Promise.allSettled(
    urls.map(fetchWithRetry)
  );
  return results.map(r =>
    r.status === "fulfilled" ? r.value : null
  );
}

🛡️ TypeScript - JavaScript עם טיפוסים

TypeScript מוסיף למערכת בדיקת טיפוסים בזמן פיתוח (לא בזמן ריצה בדפדפן) - הקומפיילר (tsc) בודק שהקוד "עקבי" מבחינת טיפוסים לפני שהוא בכלל רץ, ותופס קטגוריה שלמה של באגים (שדה חסר, טיפוס שגוי, ערך לא חוקי ב-union type) לפני שהם מגיעים לפרודקשן.

Interfaces מגדירים "צורה" של אובייקט (אילו שדות, אילו טיפוסים). Union types (כמו "admin" | "member") מגבילים ערך לרשימה סגורה של אפשרויות - ניסיון להקצות ערך אחר נתפס מיידית בקומפילציה, לא רק בזמן ריצה כשמשתמש אמיתי כבר נתקל בבאג.

Generics (function firstItem<T>(items: T[]): T) מאפשרים לכתוב פונקציה שעובדת עם כל טיפוס בלי לוותר על בדיקת טיפוסים - הקומפיילר "זוכר" איזה טיפוס נכנס ומוודא שהטיפוס שיוצא תואם, בכל קריאה בנפרד.

📸 צילום מסך אמיתי - קוד שרץ בפועל
קומפיילר TypeScript אמיתי (tsc --strict) שתופס שגיאת טיפוסים אמיתית - ערך "superadmin" לא שייך לטיפוס המוגדר.
🎯 אתגר עצמאי - נסו לפני שתציצו בפתרון
הגדירו interface בשם Product עם שדות id (מספר), name (מחרוזת), ו-status ("in_stock" | "out_of_stock"). כתבו פונקציה canOrder(product: Product): boolean שמחזירה true רק אם הסטטוס הוא "in_stock". נסו להעביר ערך סטטוס לא חוקי ובדקו שהקומפיילר תופס את זה.
רמז: union type עם שתי מחרוזות ספציפיות (לא string כללי) הוא מה שמאפשר לקומפיילר לתפוס ערכים לא חוקיים.
interface Product {
  id: number;
  name: string;
  status: "in_stock" | "out_of_stock";
}
function canOrder(product: Product): boolean {
  return product.status === "in_stock";
}

⚛️ React - Components, Props, State

React בונה ממשקים מקומפוננטות - פונקציות שמקבלות props (נתונים מההורה) ומחזירות תיאור של UI (JSX). קומפוננטה יכולה להשתמש בקומפוננטות אחרות בתוכה (Composition) - בדיוק כמו פונקציות רגילות שקוראות זו לזו.

State (באמצעות useState) הוא זיכרון פנימי לקומפוננטה שמשתנה עם הזמן ("נזכר" בין רינדורים). כשה-state משתנה, React מרנדר מחדש את הקומפוננטה אוטומטית עם הערך העדכני - זה ההבדל המרכזי מ-props, שמגיעים מבחוץ ולא משתנים מבפנים.

Hooks (פונקציות שמתחילות ב-use) הם הדרך המודרנית לתת לקומפוננטות פונקציונליות יכולות כמו state, effects, וקונטקסט - בלי לכתוב class components. חוק ברזל: hooks נקראים תמיד באותו סדר, לעולם לא בתוך if או לולאה.

📸 צילום מסך אמיתי - קוד שרץ בפועל
קומפוננטות React אמיתיות (Badge/Counter/Dashboard) עם Props, State, ו-Composition, מרונדרות בפועל ל-HTML אמיתי דרך ReactDOMServer.
🎯 אתגר עצמאי - נסו לפני שתציצו בפתרון
בנו קומפוננטת TodoList עם state של מערך משימות. הוסיפו כפתור "הוסף משימה" שמוסיף פריט חדש ל-state (לא משנה את המערך המקורי ישירות!), ורנדרו כל משימה עם .map().
רמז: setTodos([...todos, newTodo]) יוצר מערך חדש עם spread - אף פעם לא todos.push() ישירות על ה-state.
function TodoList() {
  const [todos, setTodos] = useState([]);
  const addTodo = () => {
    setTodos([...todos, { id: Date.now(), text: "חדש" }]);
  };
  return (
    <div>
      <button onClick={addTodo}>הוסף משימה</button>
      <ul>
        {todos.map(t => <li key={t.id}>{t.text}</li>)}
      </ul>
    </div>
  );
}

▲ Next.js - React ברמת פרודקשן

Next.js הוא framework שבנוי מעל React ומוסיף את מה שחסר כדי לבנות אתר אמיתי: ניתוב מבוסס-קבצים (App Router - כל תיקייה תחת app/ הופכת לנתיב URL), רינדור בצד שרת, ו-API routes בפרויקט אחד.

ההבחנה החשובה ביותר ב-App Router: Server Components (ברירת המחדל) רצים רק בשרת - יכולים לגשת ישירות למסד נתונים, לא שולחים JS מיותר לדפדפן, אבל לא יכולים להשתמש ב-useState או ב-event handlers. Client Components (מסומנים עם "use client" בראש הקובץ) רצים גם בדפדפן - נדרשים לכל אינטראקטיביות (טפסים, כפתורים, state).

API Routes (קבצי route.ts בתוך app/) מאפשרים לבנות endpoint-ים של backend באותו פרויקט Next.js - לא צריך שרת Node נפרד לפעולות פשוטות. Middleware רץ לפני כל בקשה (לדוגמה: בדיקת authentication) - נקודת בקרה מרכזית לפני שהבקשה מגיעה לדף.

🔬 מבנה תיקיות אמיתי של App Router
app/ directory structure
app/
├── layout.tsx          # שכבת עטיפה משותפת לכל הדפים
├── page.tsx             # דף הבית (/)
├── dashboard/
│   ├── page.tsx          # /dashboard (Server Component)
│   └── settings/
│       └── page.tsx      # /dashboard/settings
├── api/
│   └── users/
│       └── route.ts      # POST/GET /api/users
└── components/
    └── SearchBox.tsx     # "use client" - אינטראקטיבי
💡 שם התיקייה = חלק מה-URL, שם הקובץ (page.tsx) = מה שמוצג בנתיב הזה. אין צורך בקובץ ניתוב נפרד - זה File-based Routing.
