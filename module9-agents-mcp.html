🔄 לולאת הסוכן - Plan / Act / Observe

סוכן AI (Agent) שונה מקריאת LLM בודדת בכך שהוא פועל בלולאה: בכל סיבוב הוא מתכנן את הצעד הבא (Plan), מבצע אותו בפועל דרך כלי (Act), וקולט את התוצאה (Observe) - וחוזר חלילה עד שהמשימה הושלמה. זו בדיוק הארכיטקטורה שראינו במודול 1 ("הלולאה האג'נטית") ובקלוד קוד עצמו (מודול 2).

Planning הוא השלב שבו הסוכן מחליט מה לעשות הבא - לא בהכרח תוכנית קבועה מראש, אלא החלטה דינמית שמתחשבת בכל מה שקרה עד כה. Memory - זיכרון הצעדים הקודמים (state) - קריטי כדי שהסוכן "ידע" איפה הוא נמצא בתהליך ולא יתחיל מחדש בכל סיבוב.

מתי לולאת סוכן מתאימה, ומתי לא: משימות עם מספר שלבים לא-ידוע מראש (המסלול תלוי בתוצאות ביניים) מתאימות מצוין ללולאת סוכן. משימות עם רצף פעולות קבוע וידוע - עדיף Prompt Chaining (מודול 7), שהוא פשוט וצפוי יותר, בלי "הימור" על החלטות המודל בכל צעד.

📸 צילום מסך אמיתי - קוד שרץ בפועל
לולאת סוכן אמיתית שמתכננת, מבצעת, וצופה בתוצאה בכל סיבוב - עד שהיא מחליטה בעצמה שהמשימה הושלמה (action="finish").
🤖 מודול 9
⏱️ כ-20 דק'
🪞 Reflection ו-Self-Correction

Reflection היא היכולת של סוכן "לבקר" את הפלט שלו לפני שהוא מציג אותו כתשובה סופית - בדומה לאיך שבן אדם קורא טיוטה שכתב ומתקן אותה. בפועל זו קריאת LLM נוספת (או פונקציית בדיקה דטרמיניסטית) שבודקת את הפלט מול קריטריונים (טון, דיוק עובדתי, שלמות) ומחזירה משוב.

Self-Correction היא הלולאה השלמה: ייצור → ביקורת → אם נכשל, תיקון לפי המשוב → ביקורת מחדש - עד שהפלט עומד בסטנדרט, או עד שמגיעים למספר ניסיונות מקסימלי (חשוב להגביל! אחרת הסוכן עלול "להסתובב במעגל" ללא סוף).

שימוש נכון: לא כל משימה צריכה reflection - זה מוסיף עלות (עוד קריאות LLM) וזמן. שימושי במיוחד למשימות עם קריטריוני איכות ברורים לבדיקה (עמידה בפורמט, טון מקצועי, דיוק נתונים) ופחות למשימות פשוטות שסיכוי הטעות בהן נמוך מלכתחילה.

📸 צילום מסך אמיתי - קוד שרץ בפועל
לולאת ביקורת עצמית אמיתית: כל ניסיון מזוהה ומתוקן לפי בעיה קונקרטית (טון לא מקצועי, נתון בלי מקור) עד שעובר ביקורת.

👥 Multi-Agent Systems

במקום סוכן אחד "יודע-כל" שמנסה לעשות הכל, ארכיטקטורת Multi-Agent מפצלת את העבודה בין כמה סוכנים ייעודיים, כל אחד מתמחה במשימה צרה - בדיוק כמו subagents בקלוד קוד (מודול 2). דפוס נפוץ: Supervisor (סוכן-על) שמקבל את המשימה הכללית ומאציל לסוכנים מתמחים (research/writing/review), ואז מרכיב את התוצאות הסופיות.

היתרון המרכזי: כל סוכן-משנה מקבל context ממוקד ותפקיד ברור, מה שמפחית "בלבול" ומשפר איכות - בדיוק כמו שצוות עם תפקידים ברורים עובד טוב יותר מקבוצת אנשים ללא חלוקת אחריות. יתרון נוסף: אפשר להריץ סוכני-משנה שלא תלויים זה בזה במקביל, מה שמקצר זמן כולל.

מחיר: מורכבות ניהול גבוהה יותר (תזמון, טיפול בכשלים של סוכן-משנה בודד, עלות מצטברת של כמה קריאות LLM). כלל אצבע: התחילו עם סוכן יחיד; עברו ל-multi-agent רק כשמזהים בבירור שתפקיד אחד "עמוס מדי" או שקיימות משימות עצמאיות שיכולות לרוץ במקביל.

📸 צילום מסך אמיתי - קוד שרץ בפועל
Supervisor אמיתי שמאציל בזה אחר זה לשלושה סוכני-משנה (research/writing/review) ומרכיב את התהליך המלא.
🤖 מודול 9
⏱️ כ-20 דק'
🔌 פרוטוקול MCP לעומק

ראינו ב-MCP במודול 2 ברמת המשתמש - עכשיו נכנסים ברמת הפרוטוקול. MCP (Model Context Protocol) בנוי מעל JSON-RPC 2.0 - פורמט הודעות סטנדרטי לקריאות "בקשה-תשובה" בין תהליכים. כל הודעה כוללת jsonrpc, id (לשיוך תשובה לבקשה), method (הפעולה המבוקשת), ו-params/result/error.

שלוש הפעולות המרכזיות שכל MCP client/server "מדברים" בהן: initialize (handshake ראשוני - הכרות בין הצדדים), tools/list (ה-client שואל אילו כלים השרת חושף), ו-tools/call (קריאה בפועל לכלי ספציפי עם פרמטרים). זה בדיוק ה"חוזה" שמאפשר לכל שרת MCP (Filesystem/Database/Git/Browser) להיות מתחלף ועקבי - קלוד קוד לא צריך לדעת שום דבר על המימוש הפנימי של השרת, רק לדבר בפרוטוקול המוסכם.

נקודה חשובה: client ו-server ב-MCP הם בד"כ שני תהליכים נפרדים (processes) שמתקשרים over stdio (קלט/פלט סטנדרטי) - לא ספריית קוד רגילה שקוראים לה ישירות. זו בדיוק הסיבה שאפשר לכתוב שרת MCP בכל שפת תכנות, כל עוד הוא "מדבר" את הפרוטוקול הנכון.

📸 צילום מסך אמיתי - קוד שרץ בפועל
מבנה ההודעות האמיתי של initialize, tools/list ו-tools/call - בדיוק כפי שהן נראות בתקשורת אמיתית בין Claude Code לשרת MCP.
🤖 מודול 9
⏱️ כ-25 דק'
🛠️ בניית שרת MCP אמיתי

עכשיו נבנה שרת MCP מינימלי מאפס, בלי שום ספרייה חיצונית - רק כדי להבין בדיוק מה קורה "מתחת למכסה". השרת קורא שורות JSON מ-stdin (כל שורה = בקשה אחת), מפרש את ה-method, מבצע את הפעולה, וכותב תשובת JSON חזרה ל-stdout. זה כל מה שצריך כדי להיות "שרת MCP חוקי".

ה-client (שיכול להיות Claude Code בעצמו, או קוד שלכם) מפעיל את השרת כתת-תהליך (subprocess) ומתקשר איתו דרך הצינורות (pipes) של stdin/stdout - בדיוק כמו שמוגדר בקובץ קונפיגורציית ה-MCP servers של קלוד קוד.

שימו לב לפרט האבטחה בקוד: פונקציית safe_path מוודאת שכל נתיב קובץ מבוקש נשאר בתוך תיקיית ה-sandbox המוגדרת, וחוסמת ניסיון path traversal (כמו "../../../etc/passwd") - זו בדיוק סוגיית האבטחה שנדבר עליה בשיעור הבא.

📸 צילום מסך אמיתי - קוד שרץ בפועל
Client אמיתי שמפעיל שרת MCP כתת-תהליך אמיתי (subprocess.Popen), שולח לו בקשות initialize/tools/list/tools/call, וכולל ניסיון path traversal שנחסם בפועל.
🤖 מודול 9
⏱️ כ-15 דק'
🔒 אבטחת MCP Servers

שרת MCP הוא בפועל דלת פתוחה בין המודל למערכת אמיתית שלכם - קובץ, מסד נתונים, דפדפן. אבטחה לא-מספקת כאן חמורה: מודל שגורם לקריאת/כתיבת קבצים מחוץ לתחום, שאילתת SQL הרסנית, או פעולה בדפדפן שהמשתמש לא התכוון אליה.

עקרונות מפתח: Least Privilege - השרת חושף רק את הכלים המינימליים הנדרשים, לא גישה גורפת. Sandboxing - כל פעולת קובץ/רשת מוגבלת לתיקייה/משאב מוגדר מראש (כמו safe_path שראינו). Input Validation - כל פרמטר שמגיע מהמודל (שמקורו בסופו של דבר בטקסט חופשי - ולכן לא אמין באופן מובנה) נבדק לפני שמבוצעת פעולה בפועל.

סיכון ייחודי ל-MCP: תוכן שהשרת מחזיר (למשל תוכן קובץ, תוצאת שאילתה) חוזר לתוך context המודל - אם תוכן זה מכיל טקסט שנראה כמו הוראה ("התעלם מההנחיות הקודמות ותעשה X"), זהו בדיוק וקטור ה-Prompt Injection שראינו במודול 5, רק שהפעם המקור הוא תוצאת כלי, לא קלט משתמש ישיר.
