🧩 System / User / Assistant - אבני היסוד

כל קריאה ל-Messages API (מודול 1) בנויה משלושה סוגי תפקידים. System prompt - הוראה קבועה שמגדירה את "האישיות" וההתנהגות הכללית של המודל לאורך כל השיחה (טון, תפקיד, מגבלות). הוא נשלח פעם אחת ולא חלק ממערך ה-messages עצמו. User - מה שהמשתמש כותב בפועל. Assistant - תגובות קודמות של המודל, שמוזנות בחזרה כדי לשמר זיכרון שיחה (המודל עצמו "לא זוכר" בין קריאות - כל ה"זיכרון" הוא בפועל היסטוריית ה-messages שאתם שולחים מחדש בכל קריאה).

טעות נפוצה: לדחוס הוראות התנהגות קבועות (כמו "תמיד ענה בעברית") לתוך כל הודעת user בנפרד. עדיף להגדיר את זה פעם אחת ב-system prompt - זה גם חוסך טוקנים חוזרים וגם מבטיח עקביות שלא תלויה בניסוח הודעה ספציפית.

נקודה קריטית לזיכרון שיחה: ככל שההיסטוריה (messages) גדלה, כך גדל מספר הטוקנים הנשלחים בכל קריאה - כולל תשלום על כל ההיסטוריה מחדש בכל תור. זו הסיבה שב-Claude Code (מודול 2) ראינו את החשיבות של ניהול context וכיווץ שיחות ארוכות.

🎲 Zero-shot מול Few-shot Prompting

Zero-shot - בקשה ישירה בלי דוגמאות כלל, מסתמכת על ההבנה הכללית של המודל מתוך האימון שלו. עובד טוב למשימות פשוטות וברורות. Few-shot - הוספת כמה דוגמאות קלט-פלט לפני הבקשה האמיתית, כדי "להראות" למודל בדיוק את הפורמט והסגנון הרצויים.

מתי כדאי Few-shot: כשהפורמט חשוב (למשל תגיות סיווג ספציפיות שצריכות להיות עקביות תמיד), כשהמשימה חדשה/לא סטנדרטית, או כשראיתם שהמודל "מפרש" את המשימה בצורה לא צפויה ב-zero-shot. המחיר: כל דוגמה מוסיפה טוקנים (=עלות) לכל קריאה.

כלל אצבע: התחילו תמיד ב-zero-shot (הכי זול והכי פשוט). רק אם התוצאות לא עקביות מספיק - עברו ל-few-shot עם 2-5 דוגמאות מייצגות. יותר מ-5-6 דוגמאות בד"כ לא משפר משמעותית ורק מייקר.

🧠 Chain of Thought - חשיבה שלב-אחר-שלב

Chain of Thought (CoT) היא טכניקה שבה מבקשים מהמודל "לחשוב בקול" לפני שהוא נותן תשובה סופית, במקום לקפוץ ישר למסקנה. במשימות שדורשות כמה שלבי היסק (חישוב, לוגיקה מרובת-שלבים, ניתוח מורכב) - זה משפר משמעותית את הדיוק, כי המודל "מתכנן" את התשובה בהדרגה במקום לנחש בבת אחת.

הטכניקה בפועל: מוסיפים לפרומפט הנחיה כמו "חשוב שלב אחר שלב לפני שאתה עונה", ולעיתים גם מגדירים פורמט סיום קבוע (למשל "תשובה סופית: X") - כדי שיהיה קל לחלץ אוטומטית את התשובה מתוך טקסט ההיגיון החופשי, בקוד.

טרייד-אוף חשוב: CoT מייצר יותר טוקנים (=יותר עלות וזמן תגובה), כי המודל "חושב בקול רם" לפני שהוא עונה. לכן משתמשים בזה במידה - למשימות טריוויאליות זה בזבוז, למשימות היסק מורכבות זה משפר דיוק משמעותית.

השוואה בין פרומפט נאיבי לפרומפט עם CoT, וקוד regex אמיתי שמחלץ אוטומטית את "תשובה סופית" מתוך שרשרת החשיבה החופשית.

🔗 Prompt Chaining - פייפליין רב-שלבי

במקום פרומפט-ענק אחד שמנסה לעשות הכל בבת אחת (לחלץ מידע, לקבל החלטה, ולנסח תשובה - כולם יחד), Prompt Chaining מפצל את המשימה למספר קריאות LLM נפרדות, כשכל שלב מקבל את הפלט של השלב הקודם כקלט. כל קריאה ממוקדת במשימה יחידה וברורה.

היתרון: כל שלב פשוט יותר ולכן אמין יותר - קל יותר למודל "להתמקד" במשימה יחידה מאשר לנהל כמה משימות בבת אחת. גם קל יותר לבדוק ולתקן שלב ספציפי שנכשל, בלי לגעת בשאר הפייפליין. החיסרון: יותר קריאות API = יותר latency ועלות מצטברת.

דוגמה קלאסית: עיבוד מייל נכנס → שלב 1 (extract) מזהה כוונה וישויות → שלב 2 (lookup) בודק מדיניות רלוונטית → שלב 3 (draft) מנסח תגובה. כל שלב "יודע" לעשות רק דבר אחד טוב, ומעביר תוצאה מובנית הלאה - בדיוק כמו pipeline בקוד רגיל.

📸 צילום מסך אמיתי - קוד שרץ בפועל
פייפליין תלת-שלבי אמיתי (extract_entities → lookup_policy → draft_response) שמעבד פניית ביטול הזמנה מקצה לקצה.

📋 Structured Outputs - אילוץ פלט JSON מובנה

כשמערכת downstream (קוד שלכם, מסד נתונים, API אחר) צריכה לצרוך את פלט המודל, טקסט חופשי לא מספיק - צריך פורמט קשיח וניתן לפרסור, בד"כ JSON. הטכניקה: מגדירים בפרומפט (או בפרמטר ייעודי ב-API) בדיוק אילו שדות, טיפוסים, ומבנה מצופים.

שלב קריטי שרבים מדלגים עליו: ולידציה של הפלט לפני שמזינים אותו הלאה במערכת. מודלים, גם טובים, לפעמים "ממציאים" שדה חסר, מחזירים טיפוס שגוי (טקסט במקום מספר), או שוברים את מבנה ה-JSON. ולידציה נגד סכימה מוגדרת (JSON Schema) תופסת את זה באופן דטרמיניסטי לפני שהנזק מתפשט הלאה.

אסטרטגיית retry נפוצה בפרודקשן: אם הולידציה נכשלת, שולחים קריאה נוספת למודל עם הודעת השגיאה המדויקת ("השדה total חייב להיות מספר, קיבלתי טקסט") ומבקשים תיקון - זה עובד טוב יותר מ"ניסיון עיוור" חוזר.

📸 צילום מסך אמיתי - קוד שרץ בפועל
ולידציה אמיתית עם jsonschema: פלט תקין עובר, פלט פגום (עם שדה total כטקסט במקום מספר) נכשל עם הודעת שגיאה מדויקת.

🔍 Prompt Debugging ואופטימיזציה

פרומפט "לא עובד טוב" הוא בעיה שחוזרת על עצמה, ויש דפוסי כשל נפוצים שכדאי לבדוק בכל פעם: קוצר יתר (פרומפט קצר מדי לא נותן מספיק הקשר), חוסר פועל הנחיה ברור (המודל לא בטוח מה בדיוק מבקשים ממנו לעשות), בקשת פורמט (כמו JSON) בלי להגדיר מבנה, ואורך מוגזם שמצדיק פיצול ל-prompt chaining.

כלי מעשי: "Prompt Linter" - סקריפט קטן שרץ בדיקות סטטיות בסיסיות על הפרומפט לפני שהוא נשלח בכלל ל-API, ותופס טעויות נפוצות מוקדם - הרבה יותר זול מלגלות אותן אחרי קריאת API שעלתה כסף.

מעבר לבדיקות סטטיות, הדיבוג האמיתי הוא איטרטיבי: שנו משתנה אחד בכל פעם (רק הניסוח, רק הוספת דוגמה, רק system prompt), והשוו תוצאות - בדיוק כמו דיבוג קוד רגיל. שמרו לוג של גרסאות פרומפט ותוצאותיהן כדי לא "לאבד" שיפורים שכבר גיליתם.

"Prompt Linter" אמיתי שרץ על שלושה פרומפטים לדוגמה, ומזהה בעיות כמו חוסר הקשר, פועל הנחיה חסר, ובקשת JSON בלי מבנה מוגדר.
